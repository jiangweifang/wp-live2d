WordPress 管理画面 → Live2D 設定 → 基本設定 →直リンク防止(Cubism 4+ モデル)は V2 モデル限定のスイッチで、内部設定キーは protectV2 です。2 つの排他値があり、V2 モデルをどこに置いているかに合わせてどちらかを選んでください。両方を重ねがけしてはいけません。

V1(journey-ad / Cubism 2~3)モデルはこのスイッチを通りません。V1 モデルはプラグインのローカル REST(/wp-json/live2d/v1/model/*)から配信されているため、別途直リンク防止を被せる必要はありません。本ページは V2 のみが対象です。

オプション A:パススルー(すでにオブジェクトストレージ / CDN を使っているサイト向け)

このモードではプラグインは modelAPI をそのままブラウザに渡します。直リンク防止はオブジェクトストレージや CDN 側に任せます。よくある手段を簡単な順に 3 つ:

1. パブリック読み + Referer ホワイトリスト(最も簡単)

  • Aliyun OSS:バケット → データ保護 → 直リンク防止 で、自サイトのドメインをホワイトリストに追加。
  • Tencent COS:バケット → セキュリティ管理 → 直リンク防止設定 で同様の操作。
  • 仕組み:ブラウザがモデルを読み込む際に Referer を送るので、ホワイトリスト外のオリジンは 403 で弾かれます。
  • 限界:キャプチャに通じている人なら Referer を偽装して回避できます。日常用途には十分です。

2. 自社 CDN を被せて URL 署名を有効化

OSS の前段に Aliyun CDN / Tencent CDN / Cloudflare を挟み、「URL 署名」または「Token 検証」を有効化したうえで、署名付きアドレスを modelAPI に入力します。Referer のみより厳しいですが設定は少し手間です。

3. プライベート読み + 署名 URL(最も厳しい)

バケットを Private にし、毎回の取得に署名を要求します。このモードはプラグイン設定画面から直接構成できません。PHP に明るい人がテーマの functions.php 等で、署名 URL を動的に生成する仕組みを書く必要があります。本プラグインは特定クラウドベンダーの SDK を同梱しません。理由は下の FAQ をご覧ください。

オプション B:ローカルキャッシュ(おすすめ / 直リンク防止を設定できないソース向け)

有効化すると、プラグインが modelAPI から参照される全ファイルを WordPress 配下(wp-content/plugins/live-2d/model/)に取り込み、訪問者には暗号化された一時署名 URL のみを露出します:

  • 訪問者が DevTools で見るのもこの一時 URL で、元のオリジンは漏れません。
  • 各ファイルにはエイリアス(ハッシュ化されたパス)が付与され、第三者には推測できません。
  • 初回ダウンロード時のみソース側の帯域を消費し、以後は自サーバーから配信されます。少帯域(月 1TB 程度)のホスティングでも安心です。
  • ダウンロードはバックグラウンドの cron で実行されますが、V2 リソースへの初回フロントエンドアクセスでも遅延ダウンロードが起動します。

典型的なユースケース

  • モデルが GitHub Pages / GitHub Raw / jsDelivr にある。
  • モデルが知人のサーバーにあり、Referer ホワイトリストの追加を依頼できない。
  • 第三者の有償モデルを購入したが、公開 URL のみ提供されており、転載されたくない。
  • オブジェクトストレージの「従量課金」が暴走するのを避けたい。

設定手順

  1. まず V2 カスタムモデルガイド に従って modelAPI と modelDir を設定。
  2. 「直リンク防止(Cubism 4+ モデル)」を ローカルキャッシュ に切り替え、保存。
  3. 下に「モデルローカルキャッシュ状態」パネルが現れ、各 modelAPI / サブディレクトリのダウンロード進捗(jobId / ファイル数 / 合計バイト数)が表示されます。
  4. 初回フロントエンドアクセス、または「今すぐダウンロード」ボタン押下後、すべての job ステータスが OK になればフロントエンドはローカルコピーから配信されます。

FAQ:なぜプラグインは OSS の Private 読み出しを直接代行しないのか?

「OSS の AccessKey をプラグインに預けて、プラグイン側で署名・転送してくれればいいのでは?」とよく聞かれます。3 つの理由があります:

  1. AccessKey は DB に置けません。WordPress の設定値はプレーンテキストで保存されるため、データベースが漏洩(侵害、バックアップ流出、共有ホスティング事故…)した瞬間に OSS バケット全体が抜き取られます。モデルの直リンクなど比較になりません。
  2. クラウドごとに署名アルゴリズムが違います。Aliyun、Tencent、AWS、Cloudflare R2、MinIO それぞれ別物です。一つ実装すれば「次は ◯◯ クラウド対応を」と永遠にきりがありません。
  3. 本当に必要な人はプラグインの助けが要りません。OSS の Private 読みを構築できる人なら、設定画面より自分で数行 PHP を書く方が早く、しかも安全(資格情報は環境変数経由で、DB を経由しない)です。

自分でコードを改造して OSS 署名 URL を扱いたい場合

現在プラグインは、この経路を差し替えるための専用フックを提供していません。どうしてもやる場合は、リポジトリを fork して src/live2d-V2Api.php の get_asset() / stream_local() を改修し、自社の Private OSS パスにマッチするとき OSS の事前署名 URL に 302 リダイレクトする実装を追加してください。

需要が高まれば live2d_v2_resolve_alias といった正式フックの追加を検討するかもしれませんが、プラグイン本体に特定クラウドベンダーの SDK を取り込むことは今後もありません。

本当に運用するなら必ず:

  • 最小権限のサブアカウントを使ってください(「GetObject」のみ、特定バケット / プレフィックス限定)。
  • SecretKey を WordPress の設定項目に直接書かないでください。環境変数や専用の Secret Manager を使ってください。
  • プラグインの V2 直リンク防止 REST ルートに Referer 検証を追加(wp-config.php で明示的に有効化、live2d-V2Api.php 内のコメントを参照)。
