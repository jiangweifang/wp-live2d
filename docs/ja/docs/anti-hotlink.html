V2 モデルの直リンク防止: パススルー vs ローカルキャッシュ

WordPress 管理画面 → Live2D 設定 → 基本設定 → API タイプでモデルの読み込み元、そしてホットリンク防止を誰が担当するかが決まります。2026-05 以降、旧来の protectV2 二状態スイッチは apiType に統合され、現在は4 オプションのラジオボタンになっています。各オプションで挙動が異なります。本ページは「カスタム新版モデル」(V2 / Cubism 4+) の 2 つのオプションのみを扱います。旧版 V1 モデルにホットリンク防止は不要です。

V1（journey-ad / Cubism 2~3）モデルは本ページ対象外です。V1 モデルはプラグイン内蔵の REST エンドポイント（/wp-json/live2d/v1/model/*）から配信されるため、追加のホットリンク防止層は不要です。本ページの内容はすべて V2 モデル専用です。

4 種類の API タイプ

API タイプ モデルの所在 ホットリンク対策担当 選ぶタイミング
ローカル旧版モデル プラグインの model/ ディレクトリ(V1) 不要 22/33/Pio/Tia など V1 旧モデルを使う場合
自前デプロイ旧版モデル サイト管理者のリモート V1 PHP API クロスオリジンは自前で対応(対応しなくてもよい) fghrsh 風の V1 エンドポイントが既にある場合
カスタム新版モデル · 直接読み込み
旧 protectV2 = direct		 OSS / CDN / GitHub Pages 等のクロスオリジン位置 あなた(オブジェクトストレージ / CDN コンソールで Referer ホワイトリスト / URL 認証を設定) モデルが既に Aliyun OSS、Tencent COS、自前 CDN 等に置かれている場合
カスタム新版モデル · サイトにキャッシュ
旧 protectV2 = local、推奨		 プラグインの model/{slug}/ ディレクトリ(V2) プラグイン自身(自動 alias + 一時署名 URL) モデルが GitHub Pages / jsDelivr / 自前 nginx などホットリンク防止能力のない場所、もしくは純ローカルにある場合

オプション A:カスタム新版モデル · 直接読み込み(オブジェクトストレージ / CDN を既に使っている方向け)

このモードでは、プラグインは modelAPI に手を加えず、そのままブラウザに渡します。ホットリンク対策はオブジェクトストレージや CDN に任せます。よくある 3 つの方法(易から難へ):

1. パブリックリード + Referer ホワイトリスト(最も簡単)

  • Aliyun OSS:Bucket → データセキュリティ → ホットリンク防止で、サイトのドメインをホワイトリストに追加。
  • Tencent COS:バケット → セキュリティ管理 → 防盜連設定、操作は同様。
  • 原理:ブラウザはモデル読込時に Referer を送るため、ホワイトリスト外からは 403 で拒否されます。
  • 限界:パケットキャプチャを少し触れる人なら Referer を偽装してバイパスできます。日常用途には十分です。

2. 自前 CDN を被せて URL 認証を有効化

OSS の前段に Aliyun CDN / Tencent CDN / Cloudflare を被せ、「URL 認証」または「Token 検証」を有効化、その上で modelAPI に署名付きアドレスを設定します。Referer 単独より厳しいですが設定が少し複雑です。

3. プライベートリード + 署名リンク(最も厳格)

Bucket をプライベートに設定し、毎回署名付きでなければダウンロードできないようにします。このモードはプラグイン設定ページから直接設定できません。PHP に明るい方がテーマの functions.php でコードを書き、動的に署名 URL を生成する必要があります。本プラグインは具体的なクラウドベンダー SDK を内蔵しません。理由は下記 FAQ を参照。

オプション B:カスタム新版モデル · サイトにキャッシュ(推奨 / ホットリンク対策が組めない配信元向け)

これを選ぶと、モデルファイルがあなたの WordPress サイト(wp-content/plugins/live-2d/model/{slug}/)に保存され、外部に露出するのは暗号化された一時署名 URL のみになります:

  • 訪問者が F12 で見るのもこの一時 URL であり、配信元のアドレスは漏れません。
  • 各ファイルにエイリアス(hash 化されたパス)が振られるため、第三者には推測不可能です。
  • 2026-05 以降はサイト管理者による手動ダウンロード/アップロードのみに対応。訪問者アクセス時の「自動レイジーダウンロード」は廃止しました(古いユーザーから「不可視で制御不能」と多くフィードバックがあったため)。
  • モデルをサイトに保存する方法は 2 通りあり、共存可能です:

「サイトにキャッシュ」を選んだ後のフォーム変化

このオプションを選ぶと modelAPI 入力欄は自動的に隠れ、システムが自動で wp-content/plugins/live-2d/model/ ルートを書き込みます。サイト管理者が手動で記入する必要があるのは「モデルディレクトリ」(modelDir) リストだけで、各行に slug(= サブフォルダ名)を 1 つずつ書きます。プラグインは {root}{slug}/{slug}.model3.json という固定テンプレートで manifest URL に展開します。

例えば haru / Pio / shizuku の 3 モデルを読みたい場合、modelDir はこう書きます:

modelDir:
  - haru
  - Pio
  - shizuku

プラグインが自動展開するのは:

https://your-site.com/wp-content/plugins/live-2d/model/haru/haru.model3.json
https://your-site.com/wp-content/plugins/live-2d/model/Pio/Pio.model3.json
https://your-site.com/wp-content/plugins/live-2d/model/shizuku/shizuku.model3.json

実際に訪問者が取得する URL は /wp-json/live2d/v2/m/{token}/{alias}?e=&s= という一時署名リンクのみで、配信元が後で 404 になっても、ホットリンク防止が掛かっても訪問者は影響を受けません。

方法 A:URL からダウンロード(モデルが既に公開 URL を持つ場合)

  1. 「API タイプ」を カスタム新版モデル · サイトにキャッシュ に切り替えて保存。
  2. ページ下部に「モデルホスティング」パネルが現れ、デフォルトは「URL からダウンロード」タブ。
  3. 「ソース URL」入力欄にモデルの完全な .model3.json URL(例:https://example.com/haru/haru.model3.json)を入力し、「サイトにダウンロード」をクリック。
  4. プラグインは URL から slug(例:haru)を派生し、即座に上方の「モデルディレクトリ」リストに追加、同時にモデルファイル一式を cron キューに投入して model/haru/ へ取得します。
  5. すべての job が「キャッシュ済み」になればフロントはローカルコピーから配信されます。ページ最下部の「設定を保存」を必ずクリックして modelDir を DB に書き込んでください。

方法 B:フォルダをアップロード(モデルがローカルにしかない場合)

  1. 「API タイプ」を カスタム新版モデル · サイトにキャッシュ に切り替えて保存。
  2. 「モデルホスティング」パネルの「フォルダをアップロード」タブに切り替え、モデルディレクトリ全体(model3.json + moc3 + テクスチャ + モーション含む)を点線枠にドラッグするか、「フォルダを選択」ボタンで選びます。
  3. プラグインは相対パスのまま model/{slug}/(slug はドラッグしたルートフォルダ名)へ直列にアップロードし、進捗バーがアップロード済バイト数を反映します。
  4. アップロード完了後、slug は上方の「モデルディレクトリ」リストへ自動的に追加されます。
  5. 上部の黄色バナーにはサーバーの 1 ファイルあたりのアップロード上限(PHP upload_max_filesize 由来)が表示されます。上限超過のファイルは赤くマークされてスキップされ、PHP 側で黙ってドロップされることはありません。テクスチャが大きすぎる場合はホスティング業者に依頼するか、wp-config.php / .htaccess で upload_max_filesize と post_max_size を引き上げてください。
  6. アップロード完了後は必ずページ下部の「設定を保存」をクリックして modelDir を DB に書き込んでください。

典型的な利用シーン

  • モデルが GitHub Pages / GitHub Raw / jsDelivr に置かれている。
  • モデルが知人のサーバー上にあり、Referer ホワイトリスト追加を依頼できない。
  • サードパーティの有償モデルを購入したが、公開 URL しか提供されず、ただ乗りされたくない。
  • オブジェクトストレージの「従量課金」請求が暴走するのが心配。
  • モデルがローカルにしかない(公開 URL なし)、ドロップして終わりにしたい。

FAQ:なぜプラグインで OSS プライベートリードを直接サポートしないのか?

「OSS の AccessKey を渡して署名・転送させるようにしてほしい」という要望をよく頂きますが、3 つの理由で対応していません:

  1. AccessKey は DB に置けません。WordPress の設定値は平文保存です。DB が漏洩(ハッキング、バックアップ流出、共用ホストの事故)した瞬間に OSS バケットが丸ごと持っていかれます。モデル盗用より遥かに深刻です。
  2. 各クラウドの署名アルゴリズムが全部違います。Aliyun、Tencent、AWS、Cloudflare R2、MinIO それぞれ独自方式。1 社対応すれば「次は XX クラウドを」と際限がありません。
  3. 本当に必要な人はプラグインの助けが要りません。OSS プライベートリードを設定できる人は、設定ページをポチポチするより数行 PHP を書く方が早く、安全(認証情報を環境変数に置けて DB に入らない)です。

自分でコードを書いて OSS 署名リンクと連携したい?

現時点では専用フックは提供していません。どうしても改造したい場合はリポジトリを fork し、src/live2d-V2Api.php の get_asset() / stream_local() を編集して、プライベート OSS パスにヒットしたら OSS 事前署名アドレスへ 302 リダイレクトするように改造する必要があります。

需要が増えれば正式な live2d_v2_resolve_alias フックを追加する可能性もありますが、プラグイン本体には今後も具体的なクラウド SDK を内蔵しません。

本当に改造する場合は必ず:

  • 最小権限のサブアカウントを使う(「オブジェクト読込」権限のみ、特定の Bucket / プレフィックスに限定)。
  • SecretKey を WordPress の設定値に直接書かないでください。環境変数か専用のシークレット管理サービスを使ってください。
  • プラグインの V2 ホットリンク REST パスに Referer 検証を追加(wp-config.php で明示的に有効化、live2d-V2Api.php のコメントを参照)。
