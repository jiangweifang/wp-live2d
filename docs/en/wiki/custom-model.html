This page is only about V2 (Cubism 4+ / 5) models. If you want to use 22 / 33 / Pio / Tia type legacy models, those are V1 — set "Workshop API type" to Local or Remote workshop in the plugin settings, no payment is required, and none of the steps on this page apply. Read What are V1 and V2? first.

1. Pay first

V2 models go through the Live2D Cubism 4+ Web SDK loader. Live2D Inc. requires third-party extensible applications to sign a commercial agreement before using Cubism 4+. The plugin author has signed a long-term contract and ships the official Cubism Core under that contract, so this path bills site owners a one-off licensing fee.

  1. Sign up and pay on www.live2dweb.com.
  2. Stay signed in to the WordPress admin (this is what gives the plugin a token tied to your account so it can verify the licence with live2dweb.com).

Without payment / sign-in, the Custom entry of "Workshop API type" is disabled or asks you to sign in.

2. Prepare your V2 model files

V2 models are exported from Cubism 4+. The manifest is *.model3.json, the main file is *.moc3, plus textures, motions and expressions.

2.1 Manifest requirements

  • The manifest file name is usually {model-name}.model3.json.
  • Paths must not contain Chinese characters.
  • If your site is on https, then the manifest and every file it references (.moc3, textures, motion3.json, physics3.json, cdi3.json, …) must also be on https; otherwise the browser will refuse them as mixed content / cross-origin.
  • Cross-origin: when hosting on OSS / COS / your own CDN, configure CORS properly — usually just whitelist your site's domain as an allowed Origin.

2.2 Where should I host the model?

Sorted from most to least recommended:

  1. Object storage with hotlink protection (preferred — built-in copyright protection): Aliyun OSS, Tencent COS, Upyun, Qiniu, etc.
    • Enable referer whitelisting and only allow your own site.
    • If you put a CDN in front, make sure the CDN inherits the same hotlink rules.
  2. The plugin's own model directory (simplest, but you have to manage your own bandwidth): 
    wp-content/plugins/live-2d/model/{model-name}/{model-name}.model3.json
    If the model folder does not exist, create it. Some themes / security plugins interfere with WordPress URL rewriting; if the file does not load, check that you can open the URL directly in a browser.
  3. Let the plugin take over via "cache locally": after filling modelAPI, switch the hotlink option to "cache locally". The plugin will pull every referenced file into wp-content/plugins/live-2d/model/ on your site and only serve a temporary signed URL to visitors, so the source URL never leaks even via DevTools. See the hotlink protection page.
  4. Public static hosting such as GitHub Pages / jsDelivr: it works, but not recommended in mainland China — jsDelivr ultimately falls back to raw.githubusercontent.com, latency is unstable, and there is no hotlink protection.

3. Fill modelAPI in the admin

WordPress admin → Live2D settings → Workshop API type: choose Custom. Below it is the modelAPI input. There are two ways to fill it, picked by the trailing character:

FormTrailing charEffectmodelDir list below
① Direct single-model URLends with .jsonLoads that model3.json as is, no further path concatenation.Ignored
② Directory root + multiple modelsends with /Treated as a root directory; combined with the modelDir list below into {modelAPI}{name}/{name}.model3.json.Required, one sub-directory name per line (must match {name}.model3.json exactly).

Examples

① Direct link:

https://cdn.example.com/live2d/haru/haru.model3.json

② Directory root + multiple models:

modelAPI:  https://cdn.example.com/live2d/

modelDir:
  - haru
  - mao
  - shizuku

The plugin will resolve them at load time as:

https://cdn.example.com/live2d/haru/haru.model3.json
https://cdn.example.com/live2d/mao/mao.model3.json
https://cdn.example.com/live2d/shizuku/shizuku.model3.json

The "blink" button on the frontend will rotate through these models.

Note: with custom models the frontend "previous / next" character buttons rarely make sense; consider disabling them in the admin.

4. Texture (material) size

Keep each texture under 2048×2048. Cubism 4+ already uses fairly large textures, so stacking several oversized PNGs will easily push first-paint past 5–10 seconds. If a model exports with oversized textures, you can re-pack them inside Cubism Editor via "Embed texture atlas → resize".

5. Advanced: super-sampling (V2 only)

Plugin 2.1.3 added a super-sampling option that sharpens textures on high-DPI displays. It only takes effect for V2 (model3.json) models; the V1 path ignores this switch.

6. Security / copyright tips

  1. Make sure you own (or are licensed for) the model. Best practice: upload to object storage with referer hotlink protection plus CORS limited to your origin.
  2. The workshop currently does not accept community uploads — we cannot enforce copyright on behalf of every uploader yet.
  3. Do not put your OSS SecretKey into a WordPress option. If you must run private-read with signed URLs, put credentials into wp-config.php environment variables; see the bottom of the hotlink page.
