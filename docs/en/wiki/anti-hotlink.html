WordPress admin → Live2D settings → Basic settings → Hotlink protection (Cubism 4+ models) is a switch that only affects V2 models. Internally it is the option protectV2. It has two mutually exclusive values; pick the one that matches where you are hosting your V2 model. Do not stack both.

V1 (journey-ad / Cubism 2~3) models do not go through this switch. V1 models are already served from the plugin's own REST endpoint (/wp-json/live2d/v1/model/*) and do not need an extra hotlink layer. Everything on this page only applies to V2.

Option A: passthrough (for sites already using object storage / CDN)

In this mode the plugin does not touch your modelAPI; it hands the URL straight to the browser. Hotlink prevention is delegated to the object storage or CDN. Three common approaches, easiest to hardest:

1. Public read + referer whitelist (easiest)

  • Aliyun OSS: Bucket → Data Security → Hotlink Protection — add your site domain to the whitelist.
  • Tencent COS: Bucket → Security Management → Hotlink Settings — same idea.
  • How it works: the browser sends a Referer when loading the model; sources outside the whitelist receive a plain 403.
  • Caveat: anyone who can sniff requests can spoof the Referer. Good enough for everyday use.

2. Wrap a CDN around it and turn on URL signing

Put Aliyun CDN / Tencent CDN / Cloudflare in front of your OSS, enable "URL signing" or "Token verification", then put the signed address into modelAPI. Stricter than plain referer checking but a bit more setup.

3. Private read + signed URLs (strictest)

Make the bucket private and require a signature for every download. This mode cannot be configured directly in the plugin settings page; it requires someone comfortable with PHP to add code in the theme's functions.php that mints signed URLs on the fly. The plugin does not bundle any vendor-specific SDK; see the FAQ below for why.

Option B: cache locally (recommended / for sources that cannot do hotlink protection)

Once enabled, the plugin downloads everything referenced by modelAPI into your WordPress installation (wp-content/plugins/live-2d/model/) and only exposes an encrypted temporary signed URL to visitors:

  • Visitors only see this temporary URL in DevTools — the original origin is never leaked.
  • Each file has an opaque alias (hash-based path) that nobody can guess.
  • Source-side bandwidth is consumed only on the first download; afterwards visitors fetch from your own server. Great for low-bandwidth hosting (e.g. ~1 TB/month).
  • Downloads run via a background cron job; lazy downloads also kick in on the first frontend hit for a V2 asset.

Typical use cases

  • Model hosted on GitHub Pages / GitHub Raw / jsDelivr.
  • Model hosted on a friend's server where you cannot ask them to add a referer whitelist.
  • You bought a third-party paid model that ships only as a public URL and you do not want freeloaders.
  • You are worried about runaway "pay-per-traffic" bills on object storage.

Setup steps

  1. Configure modelAPI and modelDir as described in the V2 custom model guide.
  2. Switch "Hotlink protection (Cubism 4+ models)" to Cache locally and save.
  3. A "Local model cache status" panel appears below, listing per-modelAPI / per-sub-directory progress (jobId / file count / total bytes).
  4. After the first frontend hit (or after pressing "Download now"), wait for all jobs to reach OK; from then on the frontend serves the local copy.

FAQ: why does the plugin not just sign my OSS private-read URLs?

People often ask "why can't the plugin just take my OSS AccessKey and sign URLs for me?" Three reasons:

  1. AccessKeys must not live in the database. WordPress options are stored in plain text; if your database leaks (compromise, leaked backup, shared-hosting incident…) you lose the entire OSS bucket — far worse than someone hotlinking a model.
  2. Every cloud has its own signing algorithm. Aliyun, Tencent, AWS, Cloudflare R2, MinIO — all different. Add support for one and someone immediately asks for "please add Foo Cloud", forever.
  3. People who actually need this don't need plugin help. Anyone capable of running OSS private-read can write a few lines of PHP themselves — easier than fiddling with a settings page, and safer (credentials can come from environment variables, never the database).

Want to patch the code yourself to use signed OSS URLs?

The plugin currently does not expose a dedicated hook for intercepting this flow. If you really want to, you'll have to fork the repo and patch get_asset() / stream_local() in src/live2d-V2Api.php so that, for paths matching your private OSS prefix, they return a 302 to a pre-signed OSS URL.

If demand grows we may add a formal live2d_v2_resolve_alias hook later, but the plugin will never bundle a vendor-specific cloud SDK.

If you do go down this road, please:

  • Use a least-privilege sub-account (only "GetObject" permission, scoped to the specific bucket / prefix).
  • Never put the SecretKey directly into WordPress options; use environment variables or a dedicated secret manager.
  • Add referer verification on the V2 hotlink REST routes (enable it explicitly via wp-config.php; see the comment block in live2d-V2Api.php).
