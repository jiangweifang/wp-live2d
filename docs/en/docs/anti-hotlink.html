V2 hotlink protection: passthrough vs. cache locally

WordPress admin → Live2D Settings → Basic Settings → API Type determines where the model is loaded from and who handles hotlink protection. Since 2026-05 the legacy protectV2 two-state switch has been merged into apiType; it is now a four-option radio with distinct behaviour per option. This page only covers the two "custom V2 model" options (V2 / Cubism 4+); legacy V1 models do not need hotlink protection.

V1 models (journey-ad / Cubism 2~3) are out of scope here. V1 models are served by the plugin's built-in REST endpoint (/wp-json/live2d/v1/model/*), so no extra hotlink layer is needed. Everything below is V2-only.

Four API types at a glance

API type Where the model lives Who blocks hotlinking When to pick this
Local legacy model Plugin's model/ dir (V1) N/A Using V1 oldies like 22 / 33 / Pio / Tia
Self-hosted legacy model Your own remote V1 PHP API You handle CORS yourself (or skip it) You already run an fghrsh-style V1 endpoint
Custom V2 · direct load
old protectV2 = direct		 OSS / CDN / GitHub Pages or any cross-origin host You (set up Referer whitelist / signed URL on your bucket / CDN) Models already live on Aliyun OSS, Tencent COS, your own CDN, etc.
Custom V2 · cache locally
old protectV2 = local, recommended		 Plugin's model/{slug}/ dir (V2) Plugin (auto alias + temporary signed URL) Models live on GitHub Pages / jsDelivr / your own nginx with no hotlink defence — or only on your local disk

Option A: Custom V2 · direct load (when you already use OSS / CDN)

In this mode the plugin does not touch your modelAPI; it hands the URL straight to the browser. Hotlink defence is delegated to your bucket / CDN. Three common approaches, easy to hard:

1. Public read + Referer whitelist (easiest)

  • Aliyun OSS: Bucket → Data Security → Hotlink protection, add your domain.
  • Tencent COS: Bucket → Security → Referer settings, similar UI.
  • How it works: the browser sends a Referer; sources outside the whitelist are blocked with 403.
  • Limit: anyone who can sniff packets can spoof Referer and bypass it. Good enough for daily use.

2. Front your bucket with a CDN that supports URL signing

Put Aliyun CDN / Tencent CDN / Cloudflare in front of your OSS bucket, enable "URL authentication" or "Token check", and put the signed URL into modelAPI. Stricter than Referer alone but more setup.

3. Private read + signed link (strictest)

Set the bucket to private; every request must carry a signature. This cannot be configured from the plugin settings page — someone comfortable with PHP needs to write a snippet in your theme's functions.php that mints signed URLs on demand. The plugin does not bundle any cloud SDK; reasons in the FAQ below.

Option B: Custom V2 · cache locally (recommended / for sources without hotlink defence)

Pick this and the model files get saved into your own WordPress site (wp-content/plugins/live-2d/model/{slug}/); the only URL ever exposed to the browser is an encrypted, time-limited signed URL:

  • What visitors see in F12 is the temporary URL — the source URL never leaks.
  • Each file gets an alias (hash-style path) so it cannot be enumerated.
  • Since 2026-05 only admin-triggered manual download or upload is supported; the visitor-triggered "auto lazy download" was removed (admins reported it was invisible and uncontrollable).

What the form looks like once you pick "cache locally"

The modelAPI input is hidden automatically and the system writes it to your site's wp-content/plugins/live-2d/model/ root for you. The only field you fill in by hand is the "Model directory" (modelDir) list: one slug (= sub-folder name) per row. The plugin will expand each row into the manifest URL using the fixed template {root}{slug}/{slug}.model3.json.

For example, to load haru / Pio / shizuku, fill in modelDir like:

modelDir:
  - haru
  - Pio
  - shizuku

The plugin will expand these into:

https://your-site.com/wp-content/plugins/live-2d/model/haru/haru.model3.json
https://your-site.com/wp-content/plugins/live-2d/model/Pio/Pio.model3.json
https://your-site.com/wp-content/plugins/live-2d/model/shizuku/shizuku.model3.json

What visitors actually fetch is always the temporary signed link /wp-json/live2d/v2/m/{token}/{alias}?e=&s=; even if the source 404s or starts blocking hotlinks later, your visitors are unaffected.

Method A: Download from URL (when the model is already on a public URL)

  1. Switch "API type" to Custom V2 · cache locally and save.
  2. The "Model hosting" panel appears below, defaulting to the "Download from URL" tab.
  3. Paste the model's full .model3.json URL (e.g. https://example.com/haru/haru.model3.json) into "Source URL", click "Download to local site".
  4. The plugin derives a slug (e.g. haru), appends it to the "Model directory" list above, and queues every file for the cron worker to fetch into model/haru/.
  5. Once every job shows "cached", visitors are served from the local copy. Click "Save settings" at the bottom of the page so modelDir is persisted.

Method B: Upload a folder (model only exists locally)

  1. Switch "API type" to Custom V2 · cache locally and save.
  2. Switch to the "Upload folder" tab inside the "Model hosting" panel. Drop the whole model directory (model3.json + moc3 + textures + motions) onto the dashed box, or click "Pick a folder".
  3. The plugin uploads files sequentially to model/{slug}/ (slug = name of the dropped root folder); the progress bar reflects bytes uploaded.
  4. When upload completes, the slug is auto-appended to the "Model directory" list above.
  5. The yellow banner at the top shows the per-file upload limit your server actually allows (driven by PHP upload_max_filesize). Files larger than the limit are flagged red and skipped — no silent loss in the PHP backend. If your textures consistently exceed the limit, ask your host or raise upload_max_filesize / post_max_size in wp-config.php / .htaccess.
  6. Click "Save settings" after upload so modelDir is persisted.

Typical use cases

  • Models hosted on GitHub Pages / GitHub Raw / jsDelivr.
  • Model lives on a friend's server you cannot ask to add a Referer whitelist.
  • Paid third-party model where the seller only gives you a public URL and you don't want freeloaders.
  • Worried about runaway "pay per traffic" bills on object storage.
  • Model only exists on your own machine (no public URL) — drop and forget.

FAQ: why doesn't the plugin sign OSS private-read URLs for me?

Three reasons we get asked this often:

  1. AccessKeys do not belong in the database. WordPress option values are plaintext; one DB leak (hack, leaked backup, misconfigured shared host) and your whole OSS bucket is gone — far worse than a leaked model.
  2. Every cloud signs URLs differently. Aliyun, Tencent, AWS, Cloudflare R2, MinIO… each has its own algorithm. Add one and there's always one more "please add XYZ cloud".
  3. Anyone who can do this doesn't need plugin help. If you can wire up OSS private reads, a few lines of PHP are easier and safer than a settings UI (credentials can live in env vars, not the DB).

Want to patch the code to sign OSS URLs yourself?

There is no official hook for this today. If you must, fork the repo and edit get_asset() / stream_local() in src/live2d-V2Api.php so they 302 to a presigned OSS URL when matching your private path.

If demand grows we may add a proper live2d_v2_resolve_alias hook, but the plugin will never bundle any concrete cloud SDK.

If you really go this route, please:

  • Use a sub-account with the minimum permission (read-object only, scoped to a single bucket / prefix).
  • Never put the SecretKey directly into a WordPress option; use env vars or a dedicated secret manager.
  • Add Referer validation to the V2 hotlink REST routes (turn it on explicitly in wp-config.php; see comments in live2d-V2Api.php).
