V2 hit areas (HitArea)

This page only covers V2 (Cubism 4+ / 5) models, i.e. *.model3.json. Hit areas on V1 (journey-ad / Cubism 2~3) models use a different mechanism (based on hit_areas_custom) and are not covered here.

1. Where do hit areas live?

For a V2 model, hit areas are defined under the top-level HitAreas field in *.model3.json. Each entry has a Name and a corresponding ArtMesh ID; the browser uses Cubism Core to compute the hit box for that ArtMesh, and a tap inside that box counts as a hit on the named area.

{
  "Version": 3,
  "FileReferences": { ... },
  "HitAreas": [
    { "Name": "Head",   "Id": "HitAreaHead"   },
    { "Name": "Body",   "Id": "HitAreaBody"   }
  ]
}
  • Name: a logical name. The plugin dispatches motions based on this name. The conventional names are Head and Body, corresponding to the head and body regions.
  • Id: the ID assigned to that ArtMesh in Cubism Editor. Select the mesh and view / edit the ID in the inspector on the right.

2. HitArea → motion group mapping

The dispatch rules are the same as Live2D's official CubismWebSamples:

HitArea tappedTriggered motion groupRequired group in model3.json's Motions
HeadSwitch expression (random one)The Expressions field (not Motions)
BodyBody-tap motion groupMotions.TapBody

If your model3.json has no Expressions / TapBody, tapping that area does nothing — but it does not throw an error either.

3. How do I mark an ArtMesh as a HitArea inside Cubism Editor?

  1. Open the .cmo3 project and switch to the Modeling view.
  2. Select the ArtMesh you want to treat as a hit zone (typically the mesh that covers the head / body region).
  3. Inspector on the right → ID: note or change to a recognisable ID (e.g. HitAreaHead).
  4. When exporting .moc3, tick "Embed texture atlas" and produce the .model3.json.
  5. Open .model3.json in a text editor and add an entry to the HitAreas array, e.g. { "Name": "Head", "Id": "HitAreaHead" }. (If you ticked "include HitArea" on export, Editor writes this for you.)

4. Debugging tips

  • If a tap does nothing, first check the browser console for motion / expression load errors.
  • Hit testing is based on the ArtMesh's actual visible mesh. If you set the ArtMesh's opacity to 0 (hiding it), it still occupies the same hit box — this is the usual cause of "I'm clicking on nothing yet it still triggers".
  • Different model authors use different naming conventions: some use HitAreaHead, some head, some even name every HitArea HitArea. The plugin dispatches by Name, so make sure Name is Head or Body.

5. Further reading
